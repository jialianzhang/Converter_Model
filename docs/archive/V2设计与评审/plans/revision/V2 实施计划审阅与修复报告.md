### 🚨 1. Task 10：碳的界面浓度计算套用通用公式导致错误

**问题分析：** 在 Task 10 的 `InterfaceConcentration` 实现中，您为了代码优雅，将碳(C)强行套用了金属元素和炉渣氧化物的通用公式（式7），并在 `elements_map` 中设定 C 的渣侧氧化物传质 $F_{MOn} = 1.0$。 这在数学上会导致：

$$w([C])^* = \frac{1.0 \times 0 + F_C \cdot w([C])^b}{F_C + 1.0 \cdot E_C \cdot a_O^*}$$

**这个结果物理上是绝对错误的！** 碳的氧化产物是 $CO$ 气体，不进入炉渣，受气体分压 $P_{CO}$ 和脱碳生成速率 $G_{CO}$ 约束。根据张强模型和速率平衡方程：$J_C = F_C(w_C^b - w_C^*) = G_{CO}(P_{CO}^* - 1)$，以及 $K_C = P_{CO}^*/(w_C^* \cdot a_O^*)$。 经过严格的代数推导，**真正的碳界面浓度公式**应为：

$$w([C])^* = \frac{F_C \cdot w([C])^b + G_{CO}}{F_C + G_{CO} \cdot K_C \cdot a_O^*}$$

*注：分子多出了* $G_{CO}$*，分母的* $E_M$ *位置被* $G_{CO} \cdot K_C$ *替代。如果套用您的原代码，相当于强行忽略了脱碳生成气体对界面的压力反馈。*

**代码修复建议：** 在 `Task 10: model/interface.py` 中，必须为碳单独开辟一个计算分支：

```
    def calc_all(self, F_dict: dict, steel_b: dict, slag_b: dict,
                 E_M: dict, a_O_star: float, G_CO: float, K_C: float) -> dict:
        # ... 前面的代码 ...
        
        result = {}
        for elem, (f_m_key, f_mon_key, ox_key, n) in elements_map.items():
            F_M = F_dict.get(f_m_key, 0.0)
            w_M_b = steel_b.get(f'w([{elem}])b', 0.0)
            
            if elem == 'C':
                # 碳的专属界面动力学公式
                numerator = F_M * w_M_b + G_CO
                denominator = F_M + G_CO * K_C * a_O_star
                result['w([C])*'] = numerator / denominator if denominator > 1e-30 else 0.0
            else:
                # 其他元素的通用渣金反应公式
                F_MOn = F_dict.get(f_mon_key, 0.0)
                w_MOn_b = slag_b.get(f'w(({ox_key}))b', 0.0)
                E = E_M.get(elem, 1.0)
                result[f'w([{elem}])*'] = self.calc_w_M_star(
                    F_M, F_MOn, w_M_b, w_MOn_b, E, a_O_star, n
                )
        return result
```

### 🚨 2. Task 11：氧通量 $J_O$ 的本体活度 $a_O^b$ 定义错误

**问题分析：** 在 Task 11 的实现中，您给出了默认参数 `a_O_b = 1e-4` 并在氧通量计算中使用了 $J_O = F_O/f_O (a_O^b - a_O^*)$。 如果 $a_O^b$ 取的是**钢液本体（bulk steel）** 的氧活度，因为钢液中的溶解氧极低（通常不到 0.05%），而界面处因为顶部在吹氧，界面氧活度 $a_O^*$ 必然远大于钢液本体的 $a_O^b$。 这会导致算出来的 $J_O$ **变成负数**！意味着模型认为“氧气正在从钢水内部往界面挥发”，这显然违背了转炉吹氧的常识。

**物理纠正：** 在多相耦合模型中，驱动氧向钢渣界面传递的根本来源是**高氧化性的炉渣（即渣中的** $FeO$**）\**和顶吹氧气射流。 这里的本体氧活度\**** $a_O^b$ ***\*必须是\**与炉渣本体** $FeO$ **平衡的氧活度**，即由渣中 $FeO$ 的质量分数算出的虚拟高氧活度：

$$a_O^b = \frac{w((FeO))^b \cdot f_{FeO}}{K_{Fe}}$$

**代码修复建议：** 在调用 `calc_J_O` 时，必须动态传入由渣相 $FeO$ 决定的供氧潜能，而不能用钢液本身的溶解氧：

```
    # 在 Engine 或循环步内部调用时：
    # K_Fe 是铁氧化的平衡常数, f_FeO 是活度
    a_O_b_from_slag = (w_FeO_b * f_FeO) / K_Fe  
    J_O = oxygen_balance.calc_J_O(F_O, w_O_b_from_slag, a_O_star, f_O)
```

## 三、 边缘情况与鲁棒性微调 (Minor Tweaks)

1. **Task 7 (StaticModel) 中的负数石灰加入量风险** 在计算造渣料时：`CaO_need = R * SiO2_in_slag - CaO_from_sinter` 如果冶炼的是低硅铁水（$SiO_2$ 极少），且因热平衡需要加入了大量冷却剂（如烧结矿自带极多 $CaO$），`CaO_need` 会变成负数，导致最终输出负数的 `lime_kg`。 **建议**：在 Task 7 中添加剪裁：`lime_kg = max(0.0, CaO_need / (LIME_COMP['CaO'] / 100.0))`。
2. **Task 12 (RootSolver) 的数值求导极小值陷阱** 在 Newton 回退算法中：`df = (f(x * 1.001) - fx) / (x * 0.001)` 如果传入的猜测值 `x` 是 $10^{-8}$ 级别，`x * 0.001` 会变成 $10^{-11}$，这很容易在浮点数截断中变成 0.0，引发 `ZeroDivisionError`。 **建议**：改用固定的极小扰动步长 $\epsilon$： `df = (f(x + 1e-8) - fx) / 1e-8`，这样可以完全避免分母因 $x$ 极小而坍缩为 0 的问题。
